Durable Agent Sessions

Define an agent, then run resumable, steerable sessions.

Give an agent a prompt and a task. OpenComputer runs it on a managed runtime and records every step as a durable, streamable, resumable session.

Built into every session

Sessions are for long-running or interactive agents. For a one-shot command in a raw box, use a sandbox directly.

runtime

Self-healing runtime

  • Runtime crashes restart automatically.
  • Idle sessions hibernate and wake on the next message.
  • Hung runs stop cleanly instead of staying stuck.

sandbox

Brain / hands sandboxing

  • Brain: agent loop. Hands: files and commands.
  • Untrusted work stays contained in the hands sandbox.
  • Your model key stays in the secret store, never in a sandbox.

stream

Live streaming and steering

  • Stream events directly from the browser with a session token.
  • Reconnect from any seq without gaps.
  • Send follow-up messages without exposing your org key.

webhook

Reliable backend delivery

  • Deliver user-level events to your webhook.
  • Requests are signed with Standard Webhooks.
  • Deliveries are retried, dead-lettered, inspectable, and redeliverable.

Build a managed coding agent in three steps

The useful minimum is one reusable agent, one durable session, and one browser-safe connection to stream the event log and steer follow-up work.

1. Create an agent

An agent stores reusable configuration: name, runtime, model, prompt, and model credential.

2. Start a session

A session starts work immediately and returns a browser-safe client_token for that one session.

3. Stream and steer

Use the session token in the browser. Native EventSource resumes from the last event id after a dropped connection.

Session lifecycle

A session is the durable unit of an agent at work: an append-only event log, the pinned agent snapshot, and lifecycle status. Compute attaches to it, detaches when idle, and reattaches on the next message.

Status Meaning
queuedScheduled; a turn has not started running yet.
runningA turn is executing.
awaiting_inputA turn ended asking a question. Reply by steering.
idleDone for now, quiescent and steerable. The sandbox is hibernated.
failedThe session errored out.
archivedClosed and read-only.

Events are the system of record

Every step the agent takes is appended in order. In the events API, switch on event type; do not parse prose to decide whether work finished or failed.

Event field Use it for
typeStable discriminator: agent.message, turn.completed, tool.call, exec.completed, errors.
levelVisibility filter: user, progress, or internal.
seqMonotonic cursor for ordering and resume.
bodyTyped payload for that event.
actorWho produced the event: human, agent, or system.

Webhooks deliver committed events

Register a destination and session events are delivered as small envelopes: at least once, signed, and retried. The envelope includes the session metadata you set at create time.

Dedupe on webhook-id, which equals the event id. Verify the raw request body with a Standard Webhooks library before parsing JSON. A delivery succeeds on any HTTP 2xx.

In the PR-review agent repo, the app creates a session with GitHub routing data in metadata. When OpenComputer sends the completion webhook, that metadata is in the envelope, so the app can update the right pull request without a database, local queue, polling loop, or always-on worker. That is the point of the example: you can build an almost frontend-only, serverless app around a durable agent because OpenComputer owns the stateful session machinery. Metadata in webhooks is one example of that pattern.

Runtime tools

The claude and codex runtimes act through a fixed set of tools against the session sandbox. The agent has no direct filesystem, shell, or network of its own.

Tool What it does
bashRun a shell command in the sandbox.
readRead a file from the sandbox.
writeWrite a file to the sandbox.
lsList a directory in the sandbox.
sayEmit a user-level message. The final say is the session result.
askAsk a question and yield with needs_input. Reply by steering.

To work on a public repository, include the repo URL and branch or commit in the task text, then have the agent clone it with bash. Prepared workspaces and private-repo access are coming soon.

Architecture

Your app starts, streams, and steers a durable event-log session. The managed runtime drives the agent loop and acts through the hands sandbox. User-level events are delivered to your webhook.
