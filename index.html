Durable Agent Sessions

Background agents are mostly the system around the model.

A pull-request reviewer looks simple from the outside. GitHub sends a webhook, your app starts an agent, and a comment appears later.

The hard part is everything between those two facts: the request must return quickly, the work can take minutes, the user needs progress, duplicate webhooks should not start duplicate reviews, generated code needs a sandbox, and the final result has to reach your backend after the original process is gone.

OpenComputer Durable Agent Sessions make that long-running run the primitive. Your app creates a session with input, metadata, limits, and webhook destinations. OpenComputer records the event log, runs the agent, handles the runtime lifecycle, and calls your app back from committed session events.

The GitHub webhook is the small part.

The handler below is normal product plumbing: verify GitHub, parse the event, decide whether it should start a review, and return an HTTP response.

If the review itself lives inside that request, the design breaks quickly. The process may be killed, GitHub may retry, a model call may hang, or the user may close the tab that was showing progress.

If you do not have a session primitive, you build one.

This is the hidden infrastructure behind a background agent. None of it is specific to pull-request review, but every production app ends up needing it.

Need What you would own Why it exists
Durable work record Job table, status fields, result storage, cleanup rules The run must outlive the HTTP request and be inspectable later.
Worker lifecycle Queue, worker pool, leases, deadlines, retries, stuck-job sweeper Agent turns can take longer than a request and can fail halfway through.
Progress stream Event table, SSE endpoint, reconnect cursor, visibility filtering Users and operators need to see what the agent is doing while it works.
Callback delivery Signed webhooks, retry policy, dead letters, redelivery tools The backend still needs the result after the browser or caller is gone.
Routing state Database row mapping job ids back to repos, PRs, users, and commits A completion event is useless unless the handler knows where to apply it.
Trust boundary Sandbox isolation, credential handling, prompt-injection boundaries An agent that reviews code may clone repos, read files, and run commands.
Idempotency Delivery ids, duplicate suppression, stale-result checks Product webhooks and backend callbacks are at-least-once systems.

The PR-review demo uses OpenComputer as the run record.

The GitHub App fetches PR context, then hands the long-running part to a Durable Agent Session. The request only waits for setup: fetch files, create the session, and post a sticky "review running" comment.

What the app still owns

  • GitHub auth, installation tokens, and webhook verification.
  • PR-specific context gathering: metadata, changed files, and diff.
  • The prompt that tells the agent how to review this PR.
  • The side effect at the end: update one sticky GitHub comment.

What the session owns

  • The ordered event log and lifecycle status.
  • The managed runtime that runs the agent turn.
  • Sandboxed file and command work through hands tools.
  • The signed completion webhook sent after `turn.completed` commits.

Metadata is the callback routing store.

The app passes opaque routing data when it creates the session. OpenComputer includes that metadata in each delivery envelope, and it is also available on the session snapshot. The completion handler does not need an app-side jobs table just to remember which repository and PR a session belongs to.

A code-review agent needs hands, not only a diff in the prompt.

The demo passes a diff as context, but the task also instructs the agent to inspect the repository in the OpenComputer hands sandbox. That is the difference between a chatbot summarizing a patch and an agent that can use the surrounding code, run focused checks, and report what it actually observed.

Public repositories can be checked out directly. Private repositories need a first-class source or short-lived token handoff so GitHub installation tokens never become prompt text. That is an important product surface, not demo glue.

The resulting application shape is deliberately small.

The app is still responsible for product semantics. OpenComputer is responsible for making the background agent run durable, observable, steerable, and callback-driven.

  1. Bootstrap a reusable agent. Store the agent id in configuration. The runtime prompt, model, limits, and model credential are not recreated on every product event.
  2. Accept a product event. The GitHub App verifies the webhook, fetches PR metadata, changed files, and a diff, then creates one Durable Agent Session.
  3. Let the session run. OpenComputer records events by sequence number, runs the turn in a managed runtime, and exposes progress through the event stream.
  4. Receive a committed result. A signed `turn.completed` webhook reaches the app with session metadata. The app fetches the result and performs the product side effect.

The point is not that PR review is special.

PR review is useful because it forces the real background-agent problems into a small example: long-running work, live progress, retries, dedupe, callback routing, sandboxed code access, and credentials. Durable Agent Sessions are the API surface for that run.

The working example is a Cloudflare Worker with no database, local queue, polling loop, or always-on container for review execution. The statefulness is in the OpenComputer session.
